# 코드 컨벤션

## 문서 목적

이 문서는 Java 코드의 컨벤션 준수 여부를 비교하고 판정하기 위한 규칙 정의서이다.
각 규칙은 `규칙 ID`, `적용 범위`, `판정 기준`을 기준으로 해석하며, 필요한 경우 `위반 예시`, `준수 예시`를 보조 기준으로 사용한다.

## 공통 판정 원칙

- `위반 예시`가 있는 경우 반드시 컨벤션 위반으로 판정한다.
- `준수 예시`가 있는 경우 컨벤션을 만족하는 기준 예시로 판정한다.
- 예시와 동일하지 않은 코드라도 `판정 기준`의 의미를 만족하면 준수로 판정한다.
- 예시와 동일하지 않은 코드라도 `판정 기준`을 어기면 위반으로 판정한다.

## CODE-FLOW-001: 반복문에서는 처리 제외 대상을 먼저 건너뛴다

### 적용 범위

- Java 반복문 내부의 조건 분기
- `for`, `while`, `do-while` 반복문

### 판정 기준

- 반복문 내부에서 특정 조건의 데이터만 처리해야 하는 경우, 처리 대상이 아닌 값을 먼저 검사하고 `continue`로 건너뛴다.
- 정상 처리 로직을 불필요하게 깊은 `if` 블록 안에 넣지 않는다.
- 이 규칙은 Early Continue/Early Return 패턴을 요구한다.

### 위반 예시

```java
for (String user : users) {
    if (user != null && !user.isBlank()) {
        process(user);
    }
}
```

### 준수 예시

```java
for (String user : users) {
    if (user == null || user.isBlank()) {
        continue;
    }

    process(user);
}
```

## EXCEPTION-001: 인터페이스에 체크 예외를 선언하지 않는다

### 적용 범위

- Java 인터페이스의 메서드 선언

### 판정 기준

- 인터페이스 메서드에서 `throws`를 사용해 체크 예외를 선언하지 않는다.
- 구현체의 세부 예외 타입을 인터페이스 계약에 노출하지 않는다.

### 위반 예시

```java
public interface FileService {
    String read(String path) throws IOException;
}
```

### 준수 예시

```java
public interface FileService {
    String read(String path);
}
```

## EXCEPTION-002: 체크 예외는 언체크 예외로 변환한다

### 적용 범위

- Java 구현 클래스의 예외 처리 코드
- 체크 예외를 직접 처리하거나 전파해야 하는 코드

### 판정 기준

- 체크 예외는 구현부에서 의미 있는 `RuntimeException` 계열 예외로 변환하여 전파한다.
- 원본 예외는 변환된 예외의 cause로 보존한다.
- 단순히 체크 예외를 상위로 그대로 노출하지 않는다.

### 준수 예시

```java
try {
    ...
} catch (SQLException e) {
    throw new DatabaseAccessException(e);
}
```

## EXCEPTION-003: 의미 없는 try-catch를 사용하지 않는다

### 적용 범위

- Java 예외 처리 코드

### 판정 기준

- 예외를 그대로 다시 던지기만 하는 `try-catch`는 사용하지 않는다.
- 추가 처리, 로깅, 예외 변환, 리소스 정리 없이 `catch`에서 동일한 예외를 다시 던지면 위반으로 판정한다.

### 위반 예시

```java
try {
    service.execute();
} catch (Exception e) {
    throw e;
}
```

## METHOD-001: 하나의 메서드는 60라인을 초과하지 않는다

### 적용 범위

- Java 클래스와 인터페이스의 모든 메서드
- 생성자

### 판정 기준

- 하나의 메서드는 메서드 선언 라인부터 종료 중괄호 라인까지 60라인을 초과하지 않는다.
- 60라인을 초과하는 메서드는 역할 단위로 분리한다.
- 테스트 메서드도 동일한 기준을 적용한다.

## GENERIC-001: 제네릭 와일드카드 사용을 지양한다

### 적용 범위

- Java 제네릭 타입 선언과 메서드 시그니처

### 판정 기준

- `<?>` 와일드카드 사용을 지양한다.
- 반환 타입, 파라미터 타입, 필드 타입에 의미 없는 와일드카드를 사용하지 않는다.
- 타입을 명확히 표현할 수 있으면 구체 타입 또는 타입 파라미터를 사용한다.
- 외부 라이브러리 API 제약으로 와일드카드가 필요한 경우에만 예외로 허용한다.

## METHOD-002: 하나의 메서드 파라미터는 4개를 초과하지 않는다

### 적용 범위

- Java 메서드와 생성자의 파라미터 선언

### 판정 기준

- 하나의 메서드 또는 생성자 파라미터 수는 4개를 초과하지 않는다.
- 파라미터가 많아지는 경우 요청 DTO, 커맨드 객체, 값 객체 등으로 묶는 것을 고려한다.

## OBJECT-CREATION-001: 정적 팩터리 메서드 파라미터가 많으면 빌더 패턴을 고려한다

### 적용 범위

- 정적 팩터리 메서드를 통한 객체 생성 코드
- `of`, `from`, `create`, `newInstance` 등 객체 생성을 담당하는 정적 메서드

### 판정 기준

- 정적 팩터리 메서드의 파라미터 수가 4개를 초과하면 Lombok의 `@Builder` 어노테이션을 통한 빌더 패턴 적용을 고려한다.
- 단순 값 전달을 위해 긴 파라미터 목록을 유지하지 않는다.

## STREAM-001: stream().forEach() 사용을 지양한다

### 적용 범위

- Java Stream API 사용 코드

### 판정 기준

- `stream().forEach()` 사용을 지양한다.
- 반복 처리만 필요하면 향상된 `for`문을 우선 사용한다.
- Stream은 필터링, 매핑, 집계 등 선언적 데이터 변환이 필요한 경우에 사용한다.

## CONSTANT-001: 고정된 값은 상수로 정의한다

### 적용 범위

- Java 코드에 직접 작성된 고정 정수 값과 문자열 값

### 판정 기준

- 의미 있는 고정 값은 `final static` 상수로 정의해 사용한다.
- 동일한 정수 또는 문자열 리터럴을 여러 곳에 반복해서 직접 사용하지 않는다.
- 로그 메시지, 테스트 데이터, 한 번만 사용되는 명확한 지역 값은 예외로 허용한다.

## ENUM-001: 타입 값은 Enum으로 정의한다

### 적용 범위

- 상태, 구분, 종류, 유형 등 제한된 타입 값을 표현하는 코드

### 판정 기준

- 타입에 대한 처리는 문자열 또는 정수 상수 대신 `Enum`을 정의해 사용한다.
- 조건문에서 타입 문자열 또는 타입 숫자를 직접 비교하지 않는다.
- 외부 연동 값은 내부 `Enum`으로 변환해 처리하는 것을 우선 고려한다.

## LOOP-001: 중첩 for문은 Map 사용을 고려한다

### 적용 범위

- Java 반복문 내부에 다시 반복문이 있는 코드
- 두 컬렉션을 비교하거나 매칭하는 코드

### 판정 기준

- 중첩 `for`문으로 두 컬렉션을 반복 비교하는 경우 `Map`을 사용한 조회 구조로 변경하는 것을 고려한다.
- 키 기반 매칭이 가능한 데이터는 먼저 `Map`으로 변환해 반복 횟수를 줄인다.
- 데이터 수가 매우 작거나 순서 기반 비교가 필요한 경우는 예외로 허용한다.

## LOOP-002: 인덱스가 필요 없으면 향상된 for문을 사용한다

### 적용 범위

- Java `for` 반복문
- 배열 또는 컬렉션 순회 코드

### 판정 기준

- 데이터의 인덱스를 참조해야 하는 상황이 아니라면 향상된 `for`문을 사용한다.
- 단순 순회를 위해 인덱스 기반 `for`문을 사용하지 않는다.
- 인덱스 값 자체가 로직에 필요하거나 특정 위치 접근이 필요한 경우는 예외로 허용한다.

## CODE-FLOW-002: 중첩 구조는 3단계를 넘기지 않는다

### 적용 범위

- Java 조건문과 반복문
- `if`, `for`, `while`, `do-while`, `switch` 블록

### 판정 기준

- 조건문과 반복문 중첩은 3단계를 초과하지 않는다.
- 중첩이 깊어지는 경우 Early Return, Early Continue, 메서드 분리, 데이터 구조 변경을 고려한다.
- `if`와 `for`가 섞인 중첩도 동일하게 단계 수를 계산한다.

## CONDITION-001: 복잡한 조건은 별도 변수로 분리한다

### 적용 범위

- Java 조건식
- `if`, `else if`, `while`, 삼항 연산자 조건부

### 판정 기준

- 조건식에 포함된 조건이 3개 이상이면 의미 있는 이름의 별도 변수로 분리하는 것을 고려한다.
- `&&`, `||`로 연결된 비교식, 메서드 호출, 상태 체크는 각각 하나의 조건으로 계산한다.
- 조건 변수명은 조건식의 의도를 드러내도록 작성한다.

### 위반 예시

```java
if (user != null && user.isActive() && user.hasPermission()) {
    process(user);
}
```

### 준수 예시

```java
boolean canProcessUser = user != null && user.isActive() && user.hasPermission();

if (canProcessUser) {
    process(user);
}
```
